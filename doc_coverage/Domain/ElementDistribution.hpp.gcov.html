      65             :  *
      66             :  * \details The element distribution attempts to assign a balanced total
      67             :  * computational cost to each processor that is allowed to have `Element`s.
      68             :  * First, each `Block`'s `Element`s are ordered by their Z-curve index (see more
      69             :  * below). `Element`s are traversed in this order and assigned to CPUs in order,
      70             :  * moving onto the next CPU once the target cost per CPU is met. The target cost
      71             :  * per CPU is defined as the remaining cost to distribute divided by the
      72             :  * remaining number of CPUs to distribute to. This is an important distinction
      73             :  * from simply having one constant target cost per CPU defined as the total cost
      74             :  * divided by the total number of CPUs with elements. Since the total cost of
      75             :  * `Element`s on a processor will nearly never add up to be exactly the average
      76             :  * cost per CPU, this means that we would either have to decide to overshoot or
      77             :  * undershoot the average as we iterate over the CPUs and assign `Element`s. If
      78             :  * we overshoot the average on each processor, the final processor could have a
      79             :  * much lower cost than the rest of the processors and we run the risk of
      80             :  * overshooting so much that one or more of the requested processors don't get

      84             :  * adjusting the target cost per CPU as we finish assigning cost to previous
      85             :  * CPUs.
